# Chaos 恢复策略与告警落地计划

## 目标
- 将当前 Chaos 报告中的 `thresholdSuggestions` 与 `scenarioSummary` 从“建议输出”推进到“可执行运维动作”。
- 在不改动核心链路行为的前提下，补齐告警规则生成、恢复策略模板与自动校验闭环。

## 当前基线
- 已具备：Chaos 注入、历史趋势、风险分级、阈值建议、Markdown/JSON 双报告、CI 摘要输出。
- 待推进：将报告结果自动映射为可直接应用的告警配置与恢复动作清单。

## 开发范围
- 报告后处理模块（脚本级）：从 JSON 报告生成告警规则建议文件。
- 恢复策略模板库：按场景输出标准恢复动作（Kafka 重启、Postgres 恢复、消费暂停策略等）。
- 校验与门禁：在 CI 中增加建议有效性检查与阈值边界检查。
- 文档与运维手册：补齐“如何落地规则”“如何执行恢复动作”。

## 分阶段实施步骤

### 阶段 1：规则映射设计（先标准化，不直接改线上配置）
1. 定义统一映射模型：
   - 输入：`thresholdSuggestions` + `scenarioSummary`。
   - 输出：告警规则建议对象（名称、表达式、阈值、窗口、严重级别、说明）。
2. 明确阈值映射策略：
   - `errorRate.recommendedPct` → 错误率告警阈值。
   - `duration.recommendedMs` → 延迟告警阈值。
   - `consecutiveFailures.recommendedCount` → 连续失败告警阈值。
3. 定义风险到严重级别映射：
   - `high -> critical`，`medium -> warning`，`low -> info`。

### 阶段 2：实现告警建议产物生成
1. 扩展 Chaos 报告脚本，新增“规则建议文件”输出（JSON 或 YAML）。
2. 规则建议文件需包含：
   - 全局规则（错误率/耗时/连续失败）。
   - 场景规则（按 scenario 风险分级输出）。
3. 保持幂等：
   - 同次执行仅产生同一批次命名产物。
   - 字段顺序稳定，便于 diff 与审阅。

### 阶段 3：恢复策略模板化
1. 建立场景恢复动作模板（脚本内结构化对象）：
   - `kafka_restart`：检查 broker 健康、分区领导者、重试策略。
   - `postgres_restart`：检查连接池、慢查询、归档与维护任务状态。
   - `kafka_pause_unpause`：检查消费者 lag、分区恢复、暂停时长参数。
2. 将模板与 `riskLevel` 结合生成“操作建议段”，写入报告和 CI 摘要。
3. 为每个恢复动作增加“执行前置检查”和“执行后验证指标”。

### 阶段 4：CI 校验与门禁
1. 在 Chaos 工作流新增规则有效性校验步骤：
   - 校验建议阈值范围合法（百分比、时延、次数）。
   - 校验关键字段非空（规则名、阈值、严重级别）。
2. 新增软门禁策略：
   - 若 `highRiskScenarios` 非空，则工作流标记 warning 并在 summary 高亮。
   - 若阈值建议超边界，直接 fail。
3. 上传产物扩展：
   - 报告（md/json）+ 规则建议文件统一上传。

### 阶段 5：文档与交付
1. README 新增“告警规则建议文件”说明与字段解释。
2. 增加“恢复操作手册”章节（按场景步骤化执行）。
3. 提供示例：从一份 chaos-report.json 生成建议规则并解释如何人工审核后落地。

## 验证计划
1. 静态校验：
   - PowerShell 脚本语法解析通过。
   - 规则建议文件 JSON/YAML 可解析。
2. 单元与回归：
   - `go test ./...` 全量通过。
   - 脚本在无历史样本与有历史样本两种情况下均生成完整建议。
3. CI 验证：
   - workflow summary 正确展示建议阈值、高风险场景与规则建议文件路径。

## 风险与应对
- 风险 1：历史样本不足导致阈值不稳定。  
  应对：保留默认阈值并标记 `sampleSize`，低样本仅给 conservative 建议。
- 风险 2：建议阈值与真实业务波动不匹配。  
  应对：增加人工审核环节，先“建议模式”运行 2 周再考虑自动化应用。
- 风险 3：规则数量膨胀导致噪声告警。  
  应对：限制场景级规则触发条件，仅 `medium/high` 输出场景规则。

## 里程碑与完成定义
- M1：规则映射与产物结构确定（含样例文件）。
- M2：脚本产出规则建议 + 恢复模板建议。
- M3：CI 校验与 summary 完成，产物可追溯。
- M4：文档完成并通过一次端到端演练。

完成定义（DoD）：
- 报告、规则建议、CI 摘要三者字段一致。
- 阈值边界校验有效，异常输入可被阻断。
- 回归测试通过，且文档可指导新成员独立执行一次演练。
