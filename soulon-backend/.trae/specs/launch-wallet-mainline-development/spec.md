# 钱包主线开发启动 Spec

## Why
当前后端已通过 staging 演练与回滚验证，已具备进入钱包主线开发的基础条件。需要将钱包开发范围、交付边界与验收标准明确为可执行规格，避免边做边改造成返工。

## What Changes
- 明确钱包主线第一阶段范围：工程骨架、网络层、鉴权壳、资产与交易查询最小可用能力。
- 定义钱包侧对后端已冻结接口的对接策略与错误处理规范。
- 增加主线开发的可观测与发布前校验要求（日志、错误上报、环境配置校验）。
- 明确阶段性交付物与验收口径，确保可按里程碑推进。
- **BREAKING**：钱包 API 契约变更不得直接修改既有 v1 契约，必须通过版本升级流程。

## Impact
- Affected specs: 钱包应用骨架能力、API 对接能力、鉴权与会话管理能力、资产与交易查询能力、发布验收能力
- Affected code: wallet 前端仓库（待创建/接入）、`contracts/wallet-api-v1.json`、后端 API 文档与对接配置

## ADDED Requirements
### Requirement: 钱包主线最小可用版本
系统 SHALL 提供钱包主线第一阶段最小可用版本，覆盖基础工程能力与核心只读业务流程。

#### Scenario: 主线初始化成功
- **WHEN** 开发团队按规范初始化钱包项目
- **THEN** 项目具备统一目录结构、环境配置、启动脚本与基础页面路由

#### Scenario: 资产与交易查询可用
- **WHEN** 用户进入钱包首页并触发资产与交易列表加载
- **THEN** 系统成功调用后端冻结接口并展示结果或可读错误态

### Requirement: API 契约冻结执行
系统 SHALL 以冻结契约为唯一对接基线，禁止未版本化的破坏性改动进入主线。

#### Scenario: 非法契约变更被阻断
- **WHEN** 开发者尝试修改已冻结的 v1 契约字段
- **THEN** CI 校验失败并提示必须走版本升级流程

### Requirement: 钱包端错误处理与可观测
系统 SHALL 在网络异常、后端错误、数据不一致时提供统一错误模型、日志记录与用户提示。

#### Scenario: 后端接口异常
- **WHEN** 接口返回超时或 5xx 错误
- **THEN** 前端显示统一错误态并记录结构化错误日志用于排障

## MODIFIED Requirements
### Requirement: 集成验证与上线前门禁
在仅后端演练通过的基础上，门禁要求扩展为“后端稳定 + 钱包最小可用链路可验证”。

#### Scenario: 主线开发进入联调
- **WHEN** 钱包完成最小可用能力并接入冻结 API
- **THEN** 需要执行钱包-后端联调清单并产出可审计验证记录

## REMOVED Requirements
### Requirement: 仅以后端演练结果作为主线启动依据
**Reason**: 该标准无法覆盖钱包侧真实交互与错误处理质量。  
**Migration**: 切换为“后端演练通过 + 钱包最小可用能力验收通过”的双条件门禁。
