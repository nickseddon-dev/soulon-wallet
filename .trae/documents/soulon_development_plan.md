# Soulon Deep Chain & Soulon Wallet 完整开发规划与开发计划

本文件是 Soulon 项目的主规划文档，覆盖链端与钱包端的目标、架构、分期、交付标准与上线路线。

已知输入条件：
- `d:\soulon_wallet\plan` 为主要技术来源。
- `d:\soulon_wallet\plan2` 当前为空，不含可执行约束。

## 1. 总体目标与边界

### 1.1 项目目标
- 构建基于 Cosmos SDK 的主权应用链 Soulon Deep Chain。
- 构建适配 Soulon Deep Chain 的多端钱包 Soulon Wallet。
- 形成可演进的链上治理、资产与质押体系，并为后续 IBC 扩展预留接口。

### 1.2 交付边界
- 本期包含：链底层、核心业务模块、钱包核心能力、测试网与主网发布流程。
- 本期不包含：跨链桥托管托底机制、法币通道、中心化托管后台。

### 1.3 设计原则
- 模块优先：链端业务均以 x/ 模块方式组织。
- 可验证优先：关键状态变更与交易流程必须可复现、可追踪。
- 安全优先：签名、密钥、参数治理与升级均采用最小权限策略。

## 2. Soulon Deep Chain 架构规划

### 2.1 分层架构
- 共识与网络层：CometBFT，负责 P2P 和 BFT 共识。
- 应用状态层：Cosmos SDK 状态机，负责交易执行与状态持久化。
- 接口层：ABCI/gRPC/REST，对接钱包、浏览器与运维工具。

### 2.2 模块规划（MVP → 扩展）

| 阶段 | 模块 | 目标能力 | 依赖 |
| :--- | :--- | :--- | :--- |
| MVP | x/auth, x/bank | 账户与资产转账 | SDK 基础模块 |
| MVP | x/staking, x/distribution | 委托、奖励分发 | x/auth, x/bank |
| MVP | x/gov | 提案与投票治理 | staking 权重 |
| 扩展 | x/slashing | 验证人惩罚机制 | staking |
| 扩展 | x/params 或 runtime params | 链上参数治理 | gov |
| 扩展 | x/upgrade | 无停机升级流程 | gov, ops |

### 2.3 Keeper 与状态访问规范
- 每个模块独立 KVStore 前缀，禁止跨模块直接写存储。
- 跨模块访问通过 `app.go` 显式注入 Keeper 引用。
- 所有 Msg Handler 保持确定性，不依赖本地时钟与外部随机源。

### 2.4 创世与网络规划
- 维护创世模板：初始账户、初始供应、验证人参数、治理参数。
- 验证人准入流程：生成 `gentx`、审查、汇总、创世固化。
- 网络拓扑：至少 1 个种子节点 + 多个验证人全节点。

### 2.5 非功能指标（首期）
- 区块时间目标：2~4 秒。
- 交易最终性：单区块确认可用，多区块确认用于高价值交易。
- 可观测性：节点健康、共识轮次、mempool、出块率全量监控。

## 3. Soulon Wallet 产品与技术规划

### 3.1 钱包定位
- 面向 Soulon Deep Chain 用户的非托管钱包。
- 支持账户管理、资产转账、质押收益、治理投票。

### 3.2 密钥与签名规范
- 使用 BIP-32/39/44 生成 HD 钱包。
- 地址编码使用 Bech32，并采用 Soulon 专属 HRP。
- 默认签名模式为 SIGN_MODE_DIRECT。
- 预留 SIGN_MODE_TEXTUAL，用于硬件钱包或高风险交易二次确认。

### 3.3 钱包能力分层
- 账户层：创建/导入/导出、助记词校验、地址派生。
- 资产层：余额查询、转账构建、手续费估算与广播。
- 质押层：验证人列表、委托、撤销委托、奖励领取。
- 治理层：提案浏览、投票提交、投票记录追踪。
- 网络层：多 RPC/gRPC 端点容灾与自动重试。

### 3.4 技术栈建议
- 钱包 SDK：CosmJS（StargateClient + SigningStargateClient）。
- 客户端形态：Web 优先，后续扩展移动端壳层。
- 通信协议：优先 gRPC，REST 作为降级路径。

### 3.5 安全与风控
- 助记词和私钥仅本地加密存储，不上传服务端。
- 交易签名前强制展示收款地址、金额、手续费与 memo。
- 实现地址白名单和高风险操作二次确认开关。

## 6. 测试策略与质量门禁

### 6.1 链端测试
- 单元测试：Keeper、Msg Server、参数边界。
- 集成测试：模块联动与状态一致性。
- 压力测试：高并发转账、委托、治理投票。

### 6.2 钱包端测试
- 单元测试：交易构建、签名流程、网络错误处理。
- 端到端测试：从登录到账户操作到交易上链回执。
- 安全测试：助记词保护、签名确认、重放场景校验。

### 6.3 质量门禁
- P0 缺陷为零。
- 关键路径自动化用例全部通过。
- 链与钱包版本映射矩阵完整并经回归验证。

## 7. 风险与应对

| 风险 | 影响 | 应对策略 |
| :--- | :--- | :--- |
| 模块间状态耦合过高 | 升级困难、缺陷扩散 | 严格 Keeper 边界与接口契约 |
| 节点配置不一致 | 测试网不稳定 | 统一配置模板与启动前检查 |
| 钱包节点依赖单点 | 用户请求失败 | 多端点容灾与健康探测切换 |
| 主网参数设置不合理 | 经济模型失衡 | 先测试网参数仿真再主网上线 |
