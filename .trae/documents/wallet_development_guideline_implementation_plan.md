# 自建区块链钱包开发实施计划（基于开发指南）

## 1. 目标与边界

- 目标：将现有 Soulon Wallet 从“基础 SDK + 部署测试能力”推进到“可上线准备”的全栈钱包系统。
- 范围：移动端钱包（UI/UX）、钱包核心 SDK（密钥/交易/网络）、后端网关与索引器、安全与合规能力。
- 约束：保持分层解耦，不在业务层直接依赖平台私有实现；移动端密钥必须留在硬件安全域。

## 2. 总体分层架构落地

### 2.1 分层与职责

1. Identity Layer（密钥安全层）
   - BIP-39 助记词、BIP-32/44 派生、coin_type 配置、TEE/KeyStore/Keychain 签名桥接。
2. Business Logic Layer（交易业务层）
   - Tx Builder、Nonce 管理、Gas Estimator、序列化适配器、广播与回执跟踪。
3. Data Layer（索引数据层）
   - ETL（历史回填 + 实时追块）、Reorg 回滚、交易历史查询、通知分发。
4. Network Layer（节点交互层）
   - 多节点负载均衡、gRPC 优先通道、JSON-RPC 兼容、Light Client 能力预留。
5. UI/UX Layer（交互层）
   - 资产中心、安全中心、DApp 连接（WalletConnect/Web3 注入）。

### 2.2 代码仓结构目标

1. soulon-wallet
   - `src/core`: 密钥/签名/交易生命周期核心能力
   - `src/network`: 节点路由、重试、熔断、健康检查
   - `src/services`: 资产、转账、质押、治理、通知
   - `src/security`: PIN/生物识别策略与风控接口
   - `src/adapters`: 序列化与链协议插件（Protobuf/RLP）
2. soulon-backend（新增子项目）
   - `gateway`: gRPC/JSON-RPC 聚合网关
   - `indexer`: Kafka 消费、区块解析、Reorg 修复
   - `api`: 历史记录、资产、通知 API

## 3. 技术栈实施决策

### 3.1 移动端

1. 首选 React Native（与现有 TypeScript 代码衔接成本更低）。
2. 通过原生桥接接入：
   - Android Keystore（StrongBox 优先）
   - iOS Keychain + Secure Enclave
3. Wallet Core 集成策略：
   - 第一阶段：用于助记词、派生、地址生成一致性验证。
   - 第二阶段：用于多链扩展与签名抽象统一。

### 3.2 服务端

1. Indexer/Gateway 使用 Golang（与 Cosmos 生态工具链兼容性更高）。
2. 数据层采用 PostgreSQL + Kafka：
   - PostgreSQL：交易历史、地址索引、状态快照。
   - Kafka：区块事件流、通知事件流、重组补偿流。

### 3.3 节点通信

1. gRPC 作为主数据通道（查询、流式区块订阅）。
2. JSON-RPC 作为兼容通道（链工具与回退方案）。
3. 统一网络客户端策略：
   - 超时、重试、节点评分、降级策略标准化。

## 4. 核心模块实施步骤

### 4.1 Identity Layer 实施步骤

1. 定义 coin_type 与派生路径标准，冻结版本并写入配置中心。
2. 实现助记词生成、校验、恢复流程（12/24词 + 可选 passphrase）。
3. 抽象签名接口（secp256k1 / ed25519），平台桥接负责硬件签名。
4. 引入“不可导出私钥”约束校验与启动时自检。
5. 增加备份验证流程（随机词位回填校验）。

### 4.2 Business Logic Layer 实施步骤

1. 交易构造器重构为插件模式（Account/UTXO 可切换）。
2. Nonce 管理器实现：
   - 本地缓存 nonce
   - 链上确认纠偏
   - 并发发送串行化控制
3. Gas Estimator 实现：
   - 预估请求
   - 安全系数策略
   - 高峰期动态提价
4. 序列化适配层：
   - 先落地 Protobuf（当前链主协议）
   - 预留 RLP 扩展接口
5. 广播回执策略：
   - mempool 接收
   - 上链确认轮询
   - 失败错误码映射与可重试判断

### 4.3 Data Layer 实施步骤

1. Backfilling：
   - 按高度区间分片并发回填
   - 幂等写入数据库（唯一索引 + upsert）
2. Forward Filling：
   - 订阅新块
   - 事务解析后写入 Kafka
   - 由消费者更新 PostgreSQL
3. Reorg 处理：
   - 存储区块父哈希与确认深度
   - 检测分叉时回滚受影响高度区间
   - 回放主链数据恢复状态
4. 状态通知：
   - 交易确认事件 -> Webhook/FCM 推送
   - 失败事件 -> 重试或降级告警

### 4.4 Network Layer 实施步骤

1. 实现多节点连接池与健康检查。
2. 引入节点评分机制（延迟、错误率、最新高度偏差）。
3. 主节点故障时自动切换并记录审计日志。
4. 预研轻客户端校验：
   - 区块头同步
   - Merkle Proof 验证接口

### 4.5 UI/UX Layer 实施步骤

1. 资产中心：
   - 原生币与代币统一展示模型
   - 余额刷新策略（主动 + 推送）
2. 安全中心：
   - 防截屏策略
   - PIN 强制重验
   - 生物识别二次授权
3. DApp 连接：
   - WalletConnect v2 接入
   - Session 授权与细粒度权限提示

## 5. 里程碑拆解与交付标准

### M1（基础安全与密钥）

1. 交付：HD 派生、助记词管理、硬件签名桥接、备份验证。
2. 验收：私钥不可导出测试通过；关键路径单测覆盖率达标。

### M2（交易闭环与网络）

1. 交付：Tx Builder、Nonce 管理、Gas 估算、序列化与广播。
2. 验收：转账在 dev/testnet 稳定通过，失败场景可解释可重试。

### M3（索引器与历史）

1. 交付：Backfill + Forward + Reorg 修复 + 历史查询 API。
2. 验收：历史记录与链上结果一致；重组场景可自动恢复。

### M4（安全强化与审计）

1. 交付：生物识别、2FA、密钥与签名审计、风险地址拦截接口。
2. 验收：渗透测试问题关闭；签名可塑性与 nonce 风险项通过审计。

### M5（生态接入与上线准备）

1. 交付：DApp 连接、扫码支付、通知中心、上线回滚预案。
2. 验收：灰度发布通过；上线监控指标稳定。

## 6. 安全与合规实施清单

1. 安全基线
   - 私钥零明文落盘
   - API 密钥分级与轮换
   - 全链路敏感日志脱敏
2. 风控策略
   - 高风险地址规则引擎
   - 异常交易频率告警
   - 大额交易二次确认
3. 合规扩展
   - KYC/AML 接口预留
   - 交易审计报表导出接口

## 7. 测试与发布策略

1. 测试层次
   - 单元测试：密钥派生、交易序列化、错误映射
   - 集成测试：钱包 SDK 与本地链联调
   - 端到端测试：移动端关键交易流程
2. 发布流程
   - 本地部署测试 -> 测试网回归 -> 预发布灰度 -> 正式发布
3. 回滚策略
   - 网关与索引器蓝绿部署
   - 客户端特性开关控制高风险功能

## 8. 与当前仓库对齐的首批执行顺序

1. 优先完善 `soulon-wallet`：
   - 补齐 Identity Layer 抽象与硬件签名接口。
   - 增强 Nonce/Gas/广播失败可重试策略。
2. 新建 `soulon-backend`（网关 + 索引器）最小可用版本。
3. 将现有 `deploy:test` 扩展为“链端启动 + 钱包在线冒烟”一体化流水线。
4. 补齐审计前自查脚本（密钥、日志、配置暴露扫描）。

## 9. 执行完成定义（DoD）

1. 钱包核心链路：创建钱包、备份校验、转账、质押、治理投票全部通过测试。
2. 索引链路：历史回填与实时追块稳定运行，Reorg 恢复验证通过。
3. 安全链路：私钥不可导出、签名安全、风控策略、日志脱敏全部达标。
4. 部署链路：一键部署测试可重复执行，失败可定位可恢复。
